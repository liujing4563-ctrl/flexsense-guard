# 证据管理规范

## 原则

结果是否进入 Git 与结果是否必须留存是两件事。大型时间序列、构建产物和临时
日志可以保存在外部受控存储；机器可读汇总、证据索引、失败记录和阶段决定必须
进入版本控制。

## 目录

```text
results/
├── p1/             # P1 汇总、索引和决定
├── p2/             # P2 汇总、索引和决定
├── p3/             # P3 汇总、索引和决定
├── integration/    # 跨语言、SIL、App 和演示验收
├── reports/        # 可审查的报告产物
└── generated/      # 本地临时生成数据，默认不提交
```

`06_validation/reports/**` 保存报告生成逻辑和模板；`results/reports/**` 保存由真实
证据生成、可追溯的报告产物，两者不得混用。

## 每次运行必需元数据

- 阶段、案例和运行唯一标识；
- Git 提交、工作区是否干净、算法版本和配置版本；
- 随机种子、软件版本、运行平台和执行时间；
- Plant 真实参数、Observer 名义参数及失配比例；
- 数据来源、调试/验证/测试划分；
- 实际命令、运行状态、原始证据位置和校验值；
- 指标窗口、汇总方法、门限和 `PASS/FAIL/NOT_VERIFIED` 决定；
- 负责同学、审查同学、已知风险和失败分类。

## 索引与命名

建议运行标识：`<stage>-<scenario>-seed<seed>-<UTC时间>`。每个阶段目录至少保存：

```text
README.md
evidence_index.json
summary.json
decision.md
```

在阶段尚未执行时只保留 README，不创建虚构的汇总或决定文件。

## 有效性规则

- JSON 不得包含 NaN、Inf 或未声明字段；
- 不删除失败案例，不覆盖历史结果；
- 修改阈值后必须创建新运行，不得回写旧决定；
- Mock 样例只能位于 `configs/mock/**`，不得复制到正式结果目录；
- 无法访问原始证据或无法复现命令时，阶段结论为 `NOT_VERIFIED`；
- 项目负责人审查证据有效性，模块负责同学对原始结果真实性负责。
