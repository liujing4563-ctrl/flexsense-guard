# Git 协作流程

## 分支模型

`main` 只接收经过人工审查的 PR。不创建长期 `develop` 分支；每项工作建立一个
短生命周期分支。

| 前缀 | 用途 | 示例 |
|---|---|---|
| `chore/` | 仓库治理和基础工程 | `chore/project-lead-foundation` |
| `feat/` | 独立功能 | `feat/dual-inertia-plant` |
| `fix/` | 已确认缺陷 | `fix/observer-input-consistency` |
| `docs/` | 仅文档 | `docs/experiment-protocol` |
| `test/` | 测试和验证工具 | `test/schema-contracts` |
| `probe/` | 技术探针 | `probe/p1-matlab-recovery` |

一个分支和 PR 只对应一个阶段目标。

## 阶段顺序

| 阶段 | 目标 | 主要负责同学 | 前置门禁 |
|---|---|---|---|
| PR-0 | 项目负责人基础工作包、模型与输入契约冻结 | 项目负责人同学，Simulink 同学共同确认 | 无 |
| PR-1 | Plant 输入修复与 P1 MATLAB 复验 | Simulink 同学 | 模型和输入契约共同确认 |
| PR-2 | P2 数据与分类基线 | 深度学习通感算同学 | P1 通过 |
| PR-3 | P3 可信失效验证 | 项目负责人同学及相关模块同学 | P1 通过 |
| PR-4 | 控制与集成 | 对应模块同学 | 所需探针通过 |
| PR-5 | C/SIL、App 和报告 | Linux、软件同学 | 主算法冻结 |

阶段门失败时先记录证据和路线决定，不继续创建后续功能 PR。

## 提交流程

1. 更新本地 `main` 并确认工作区状态。
2. 从最新 `main` 创建短生命周期分支。
3. 只修改当前目标允许的文件。
4. 运行对应测试并记录 `PASSED`、`FAILED` 或 `NOT RUN`。
5. 检查差异中没有敏感材料、真值泄漏或无关改动。
6. 使用 Conventional Commits 创建聚焦提交。
7. 推送分支并创建 Draft PR。
8. 由模块负责同学和至少一位其他同学审查。
9. 项目负责人同学完成最终集成审查。

## PR 必填内容

- 本 PR 的唯一目标和非目标；
- 修改文件和公共接口影响；
- 实际执行的命令及结果；
- 未运行检查和原因；
- 阶段门状态、风险和降级路线；
- 模块负责同学确认；
- 无真值泄漏、无虚假结果和无敏感材料声明。

禁止自动合并。未完成审查或仍有阻断问题时保持 Draft。

## 公共接口变更

公共接口变更必须在独立 PR 中同步文档、Schema、Python 类型和受影响语言定义。
不得在接口 PR 中顺带修改 Plant、Observer、分类或控制主体算法。

## 失败和回退

- 不删除失败记录；
- 修复必须引用原失败证据；
- 未运行检查不能写成通过；
- 需要回退时优先使用新的修复提交，不改写已共享历史；
- 路线降级必须更新章程、门禁、风险和 Issue 范围。
